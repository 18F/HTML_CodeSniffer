Success Criterion 1.3.1: Info and Relationships (Level A)

Information, structure, and relationships conveyed through presentation can be programmatically determined or are available in text.

(From Understanding WCAG 2.0: Understanding Success Criterion 1.3.1)

Discussion

Associating table headers with data cells

The WCAG 2.0 techniques document provides two methods of associating headers with data cells: the headers attribute and the scope attribute.

The first method requires that headers be assigned an id, and that ordinary (td) cells use a "headers" attribute which specifies the IDs of all headers it is associated with. Alternatively, the "scope" attribute is placed on a th element to state which dimension it is a header for (row, col, rowgroup or colgroup) - the group headers are not well accessibility-supported, which limits its usefulness beyond a single row or column header.

HTML_CodeSniffer recognises both methods, but which ones it allows or requires depends on the number of rows or columns with heading (th) elements:

  • With one row or one column of headers, but not both, either technique is encouraged but optional.
  • With one row AND one column of headers, one of the two techniques MUST be used, but either is acceptable.
  • With at least two rows OR two columns of headers, the headers attribute technique MUST be used.

Further, when the headers attribute technique is used and th elements are properly identified with IDs, HTML_CodeSniffer will calculate the expected value of the headers attribute for each cell, and emit an error for each cell that fails this comparison. Similar to the class attribute, order is not important: if HTML_CodeSniffer expects headers of "row1 colB" and the cell as "colB row1", this is fine.

Adding summary and caption to tables

Relevant techniques are H39: Using caption elements to associate data table captions with data tables, and H73: Using the summary attribute of the table element to give an overview of data tables.

The caption element is reasonably obvious: it adds a title to the table for identification purposes, which all users can see. On the other hand, the summary is not shown (at all) to sighted users, so the main intent is to explain the use of the table where it is not obvious to users requiring accessibility support. The W3C article provides the example of a bus timetable, which uses the summary attribute to explain how to read it (in this case: bus stops listed in row 1, then look down that column to find departure times). For sighted users reading the timetable would be obvious, but someone with a screen reader who cannot skim-read the table first would not have this orientation.

When both caption and summary are specified, the information should not be duplicated.

Labelling form elements

Labels must be explicitly labelled, using the "for" attribute, according to H44: Using label elements to associate text labels with form controls. If the visual design cannot accommodate a label, then you can use the title attribute instead - see H65: Using the title attribute to identify form controls when the label element cannot be used.

Implicit labels are where the label attribute wraps around the field: they are not supported by assistive technology as well as explicit labels, so are not recommended.

Using the label element is preferred whenever possible, because sighted users relying upon the keyboard for navigation may not be able to access the title attribute, which normally appears as a tooltip in the browser.

Grouping form elements

Using the fieldset element

Fields that are logically related should be grouped using the fieldset element, according to technique H71: Providing a description for groups of form controls using fieldset and legend elements.

The W3C acknowledges that fieldset elements are tacky with default styling:

Authors sometimes avoid using the fieldset element because of the default display in the browser, which draws a border around the grouped controls. This visual grouping is also useful and authors should seriously consider retaining it (or some form of visual grouping). The visual effect can be modified in CSS by overriding the "border" property of the fieldset and the "position" property of the legend.

This also applies when one or more fields represents a single data item: for instance, a phone number or date. This would allow a user with the appropriate assistive technology to properly skip the entire item of data in one hit. As an example, a credit card expiry date is commonly represented as two drop-down fields: month and year. Each individual field should be labelled for their purpose (eg. "Month"), probably using the title attribute because the designer is not likely to want the label to show to sighted users. The two fields should then be wrapped in a fieldset element, stating their overall purpose (eg. "Expiry Date") in the legend element.

Indeed, one common failure listed for Success Criterion 3.3.2 (Labels or Instructions) specifically relates to formatting a phone number as multiple fields, but without including a text label. (Note that this suggests three fields as per the United States' common format, but the same would apply in Australia where phone numbers are often broken up into two fields, for area code and local number.)

Using optgroup in a select element

Related options should be grouped using an optgroup element, as per technique H85: Using OPTGROUP to group OPTION elements inside a SELECT.

Using list markup

The relevant technique is H48: Using ol, ul and dl for lists or groups of links. Most checks (especially for a definition list) will likely need visual confirmation that the block should be a list, however HTML_CodeSniffer can help with certain paragraphs that should be turned into lists.

Paragraphs broken up by line breaks (br elements) will be detected as being potential lists when at least one line starts with the following:

  • an asterisk or dash, indicating an unordered list, or
  • a number followed by spacing, possibly including a punctuation mark between the two (eg. "1." or "1/" or similar), indicating an ordered list.

HTML_CodeSniffer will suggest that these elements should be converted into the appropriate list type.
