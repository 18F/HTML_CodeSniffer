Success Criterion 4.1.1: Parsing (Level A)

In content implemented using markup languages, elements have complete start and end tags, elements are nested according to their specifications, elements do not contain duplicate attributes, and any IDs are unique, except where the specifications allow these features.

Note: Start and end tags that are missing a critical character in their formation, such as a closing angle bracket or a mismatched attribute value quotation mark are not complete.

(From Understanding WCAG 2.0: Understanding Success Criterion 4.1.1)

Discussion

Valid vs. Well-formed

Keep in mind that "valid" and "well-formed" are two different things, at least in XML. In an XML document (including XHTML), it only needs to be well-formed to satisfy this Success Criterion - not necessarily valid (technique H75: Ensuring that Web pages are well-formed).

Replacements for well-formedness requirements in HTML

Because HTML serialisation does not have a concept of being "well-formed" so to speak, following technique H88: Using HTML according to spec is sufficient. Otherwise, at an absolute minimum, the following three techniques must be followed:

  • H74: Ensuring that opening and closing tags are used according to specification
  • H93: Ensuring that id attributes are unique on a Web page
  • H94: Ensuring that elements do not contain duplicate attributes

Note that technique H74 does not necessarily mean that all tags must be closed - only where they are required according to the appropriate specification. For example, a document written as XHTML 1.0 should have all tags closed, however documents written as HTML 4.01, or as HTML5 in the HTML serialisation, are allowed to omit certain closing tags - both for empty/void elements such as "br", and certain other elements such as "p" and "li" - and still satisfy technique H74.

In contrast to HTML_CodeSniffer cannot check most of these errors, because it performs its check through information it receives from the Document Object Model (DOM). Browsers generally try to correct any invalid nesting, duplicate attributes and the like, so by the time HTML_CodeSniffer gets to it, such analysis is not possible, because the DOM has been "cleaned". Further, if an XHTML document parsed as XML (rather than HTML as in most browsers) fails these techniques, a conforming parser would likely not load the document at all (as it is not well-formed), making analysis impossible.

HTML_CodeSniffer will check for duplicate ID attributes, however. These errors are flagged as a Common Failure - Failure of Success Criterion 4.1.1 due to duplicate values of type ID. The failure is defined this way because the failure is generic, and attributes can be defined as "type ID" for other types of XML documents through a document type definition (DTD) or XML Schema.

A lack of failures of this kind can only be considered authoritative if a complete document is processed. A document fragment can be tested for duplicate IDs, but it does not guarantee conformance if a fragment is inserted into a document, as it may duplicate IDs used by the fragment.
