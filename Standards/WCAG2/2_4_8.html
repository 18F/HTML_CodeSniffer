WCAG 2.0 Success Criterion 2.4.8: Location (Level AAA)

Back to: Home - Level AAA Summary

Definition

Information about the user's location within a set of Web pages is available.

More discussion from W3C: Understanding Success Criterion 2.4.8

About the link element

Using the link element to programmatically relate pages is covered in technique H59: Using the link element and navigation tools. Some user agents provide methods for using these links, but this has suffered a little due to lack of consensus on some standard link relation types.

The link element includes several different relation types or roles. Beyond stylesheet and "alternate" representations (for RSS/Atom feeds, for instance), some relations are used for navigation purposes.

Appropriate lists of relation types include:

  • The XHTML vocabulary which defines roles as they relate to XHTML+RDFa, for XHTML 1.1+.
  • The IANA link type registry as it applies to RFC5988: Web Linking. This proposes that links would be able to be expressed as HTTP headers.
  • The HTML5 specification contains a base list of supported relation types - note that only "next" and "prev" are explicitly included for navigation purposes; other proposed link types in HTML5 are explicitly the responsibility of a HTML5 link type registry maintained by the Microformats community.

Although older browsers may have displayed navigational link elements in their user interface "chrome", it is no longer a popular option, and has been removed from later browsers. Some support add-ons which add this support back in. In summary:

  • Opera 11 includes a "Fast Forward" button by default using the "next" relation; there is a "Navigation Bar" toolbar which displays the other relations, but it is hidden by default, and is likely to be removed "in due course".
  • "Old" Mozilla/Netscape used to handle it, and so does SeaMonkey now (View menu - Show/Hide - Website Navigation Bar).
  • Firefox (and probably IE) have extensions that expose link relations in a separate toolbar.

The "next" relation is also used for link prefetching in Firefox (and probably later versions of SeaMonkey), as well as the explicitly specified "prefetch" relation.

HTML5 only supports "forward" links (using the rel attribute); "reverse" links (using the rev attribute) are not supported. The only major reverse relation that was commonly used in HTML was rev="made", which referred to the author of the page - this has been changed to rel="author" in HTML5. Therefore, HTML_CodeSniffer will not recognise link elements with reverse relations.

HTML_CodeSniffer will test ALL link elements for validity, not just those specified for navigational techniques.

Additional sufficient techniques

Beyond the link element, the following techniques are also sufficient for allowing a user to determine the location of a page, where appropriate:

  • Providing a breadcrumb trail.
  • Providing a site map.
  • Indicating current location within navigation bars - but, just changing the background colour is not sufficient (see Success Criterion 1.4.1 (Use of Colour)): something else should change, such as disabling the link, bolding the text, or adding an indicator such as an asterisk or bullet.
