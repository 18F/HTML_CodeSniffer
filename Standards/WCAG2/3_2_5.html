WCAG 2.0 Success Criterion 3.2.5: Change on Request (Level AAA)

Back to: Home - Level AAA Summary

Definition

Changes of context are initiated only by user request or a mechanism is available to turn off such changes.

More discussion from W3C: Understanding Success Criterion 3.2.5

About opening links in a new window

When links are opened in a new window, the user ought to know about it before they select it. One method of doing this is to use technique H83: Using the target attribute to open a new window on user request and indicating this in link text.

Not all HTML document types officially support the target attribute - notably, it was removed from HTML 4.0 Strict and XHTML 1.0 Strict. Users should always be in control of where their next link is placed, and then-current generations of user agents did not handle these well (before the inclusion of, for instance, in-built popup blockers). However, it has been reinstated into HTML5. "Use of the target attribute provides an unambiguously machine-readable indication that a new window will open", and modern user agents can deny or modify this behaviour (such as placing it into another tab instead, or blocking entirely).

Where a link is specified using target="_blank", HTML_CodeSniffer will test for whether the link mentions that it will open in a new window. If the words "new window" (not case-sensitive) aren't found in the link name, a warning will be generated.

Since this is a AAA-level test, the fact that something opens in a new window should be part of the link title, and not part of the context - ie. the surrounding sentence or list item. This is because Success Criterion 2.4.9 would also apply, which requires the link text to describe the purpose without assistance from its context.

Where target is not supported, other methods would include technique SCR24: Using progressive enhancement to open new windows on user request. HTML_CodeSniffer cannot detect links which open up a new window using JavaScript.
